## 1. 选稿引擎 events(含根因诊断) ✅

> **人话**:换掉了"按谁最像兴趣词、谁星最多"挑新闻的烂脑子,改成"按今天哪件事最热、最新、多少家在报"挑。

### 要解决什么问题
旧选稿核四个症状(①不新鲜 ②同事件重复刷屏 ③抓不到当下最热 ④GitHub 榜全是年度老仓)其实是**同一个设计取向在管线不同位置反复犯案**:整条管线用"和一句静态兴趣词的语义相似度"当选稿主轴和硬召回闸,而新闻日报真正的活儿是"今天什么事最重要、最新"。

### 根因诊断(三共因 + 一元因,2026-06-17 多智能体勘察 + 对抗证伪)
- **共因 A — 选稿主轴选错对象**:唯一硬闸是语义余弦召回(`recall_by_vector` `ORDER BY distance ASC LIMIT 120`),新闻头条对兴趣词相似度天然只 ~0.07–0.28、arxiv 论文却 0.35–0.63 → 真炸的大新闻在召回阶段就被论文挤出去(库里抓到了,是召回机制漏的)。"多源齐报"这个最强的当天热度信号,数据里明明有(Karpathy 14 源),全管线没人当主排序轴用。
- **共因 B — "同一件事合并"在架构里无人负责**:去重层 `0.88` 阈值只合"同一篇近重复"、不合"同一事件不同报道";跨板块去重只存在于 GitHub 榜,AI/bio/geo 三域各跑各的、互相看不见 → 跨板块同事件重复天经地义。
- **共因 C — 新鲜度是"软权重"不是"硬窗"**:recency 权重对终分最大影响只 `0.2×0.4=0.08`,压不动相似度更高的旧条;5 天前旧条 recency 还有 0.3~0.64,不仅没惩罚还领分。
- **元因 — 没有测量闭环**:`freshness`/`hot_news` 两维都"未考·仅人眼",`grade_reference_topics` 空 topics 直接 `return passed=True`(空壳)→ 四症状烂了几周没警报。
- 🔴 **5 个假病因明令不许蹚**(已对抗证伪):别调 `weights.event=0.1`(瓶颈在召回不在终分权重)/ embedder query-passage 任务非根因 / r/Anthropic 重复源不造重复卡 / arxiv 非独立根因 / `freshness_hours=24 归零`是误传(各源实为 72~504h)。

### 核心设计决策 + 为什么(五根柱子)
- **柱① 事件成为一等公民**:抓进来先全局聚类成"事件"。承重墙修正——聚类**不用裸标题更不用正文**(实测正文让同新闻向量发散),改**复用 threads 的「A 候选 + B 判官」两段式**(LLM 主体短语做宽松候选召回 + LLM 判官裁决是否同一件事),不靠裸阈值硬合。🔴 生产铁规:**必须喂清洗后正文**(去 HTML/URL/base64),脏正文把判官 recall 卡死(0.849→换干净正文 0.868 过线)。
- **柱② 热度当主轴、相关性降成闸、新鲜度做硬窗**(正交轴不是加权和):热度/速度=主排序轴(按源权威加权的不同可信源数 + 加速度);相关性=只决定"够不够沾边进板块"的闸,余弦彻底退出排序;新鲜度=硬窗(主峰超 48h 直接出局,除非今天有实质更新);量级地板=大厂/官方一手首发给热度地板分(但收紧为白名单实体 + 仍需最低相关 + bonus 有上限,防"啥都贴地板分刷屏")。
- **柱③ 单一全局事件池 → 再分配到板块**:全局只聚一次,一件事只分配给一个板块、只出一张卡,把"跨板块/GitHub 榜内部/今日卡↔GitHub 榜/三板块互串"四种重复一锅端(用户 2026-06-17 四种全勾"算重复要杀")。
- **柱④ 一事件一卡,接口零改动**:每个事件选一个代表成员(源权威最高/正文最全)喂 summarize,**不碰** summarize/verify/数字 `{Fn}` 回源任何逻辑——只改"选哪条来 summarize"。数字回源命根子天然保住。
- **柱⑤ 自评闭环长在架构里**:每天独立抓"今天真该上的"(GitHub 官方 trending + AI 聚合)当真值自动算命中率,掉了报警;堵掉 `grade_reference_topics` 空壳(空 topics 不许返 passed)。记分牌**必须最先做**,否则调排序=蒙眼开车。
- **数据模型**:新增 events / event_members / event_heat_trace / repo_key / repo_timeline **五张表全 additive、只新增不改旧表**(threads/thread_clusters/item_timeline PK 一律不动),downgrade 直接 drop 零有损。`event_id` 用稳定 serial/UUID(不用成员 hash,否则身份漂移把记分牌/徽标/A-B 全搞乱)。
- **验收红线**:数字回源 100% 可回源 0 漏 token / 同事件重复对数 = 0 / 陈旧率 ≤10% 中位龄 ≤24h / 热点 Recall@10 ≥70% / GitHub 新仓 ≥50% 且 0 山寨顶替。用户最高主观尺(逐字不许软化):**"我只看到结果。我只需要看到 pulsewire 抓取的新闻够热门、够新、够前沿、够重磅。"**

### 现状
✅ `config.yaml` `rank.engine=events`,2026-06-19 切上线;治好"抓不到最热/全年度大名/同事件重复"。旧 `legacy` 引擎留一行回滚网(`rank.engine`),暂别拆。Phase 1a 聚类校准门(误合 precision=1.000 / 漏合 recall=0.868,双红线清)+ Phase 1b 真机漏斗(坐实热点死在召回+排序、不死在分类暗门)均 2026-06-17 通过。

## 2. 重磅度闸 magnitude ✅

> **人话**:events 选完稿、写榜前加一道"水货筛子",把"一眼就水"的论坛闲聊/花边踢掉,让真新闻补进前十。

### 要解决什么问题
events 引擎治好了"抓不到最热/重复/陈旧",但 `/eval` 唯一没过的"够热门"还差一口气:前十偶尔混进"有正文、够新、够相关、但就是不值得占版面"的条目(论坛闲聊/冷知识花边/纯表态),拉低"够重磅"观感。`min_body_chars` 是结构护栏(有没有真正文),重磅度闸是它的语义兄弟(这正文讲的是不是真大事)。

### 核心设计决策 + 为什么
- **力度锁死 = 真 B 档**:用户看 11 天 525 张真卡 A/B 后签字——**只砍 ①论坛/个人闲聊 + ②纯花边冷知识**;③分析/预测/评论/智库报告、④综述/科普/前沿研究解读、⑤营销稿**一律留**,宁错放别错杀。决策依据:砍"所有观点/分析/预测"会误杀深源科普(细胞力学/合成生物学等 Nature 级前沿)→ 直接违 v2「深源进、大白话出」红线。
- **插点 = 分板块之后、限额(`apply_event_quotas`)之前**,且只判每板块 heat 头部 N 条(有界 LLM 调用,不判全池几百事件)。
- **回填靠 quotas 天然实现,不手动回填**:水货只从 board_evs 删掉,quotas 本就按 heat 降序贪心推进到 final_limit,#4 自然顶上。🔴 **绝不把 board_evs 截断成"头部 N 窗口"再传**(会破坏同事件去重/源族折叠/老项限额的跨整池连锁)。
- 🔴 **B 档命门:严格真值判定 `is_water = out.get("water") is True`**——只认严格布尔 True;`{"water":"yes"}` 这类字符串字段缺失一律 fallback KEEP。绝不用 `bool(out.get("water"))` 强转(`bool("yes")==True` 会误杀真新闻;这正是去重判官里对去重无害、对本闸=误杀的口子)。
- **fail-safe 铁律**:LLM 失败/无 key/超成本闸/拿不准 → 一律判 KEEP(留),与去重判官"宁可漏合也别误合"同一套保守哲学。被砍清单打 `magnitude.water_dropped` 结构化日志(headline + reason),否则 A/B 验"零误杀"无从做。
- **配置 additive 默认关**:`magnitude_gate_enabled`(默认 False)+ `magnitude_judge_top_n`(默认 25,须 ≥ `final_limit+5`,否则第 16-20 名水货漏判)+ `max_water_judges_per_run`(全局硬顶)。回滚 = 改 `false`,闸整条短路零风险。

### 现状
✅ 已上线 `config.yaml` `magnitude_gate_enabled: true`(commit 867f09a)。离线 A/B + 独立盲考官 30 样本**误杀 0**(B 档命门 PASS)。力度演进:初版砍 18% 且误杀深源 → 真 B 档收紧砍 2.5% 全是论坛闲聊。
**2026-06-25:判官改 3 次多数票**(`magnitude_judge_votes=3`,需 2 票才砍)——治 flash 抽风:单判时偶发返错就漏(实测一条 Reddit 个人吐槽客服连判 6 次=1 假 5 真,06:00 单判抽到那次假就漏进 AI 板)。多数票既补漏、又因"单条假阳要 2 票才误杀"比单判更不易误杀;提前停(够多数/凑不够即停)~2 次/条。独立考官 PASS(真水货召回 9/9、真新闻误杀 0/9)。`votes=1` 回退旧单判。

## 2.5 话题闸 topic ✅

> **人话**:分版编辑——判每条新闻"主题属不属于这个板块",把跨领域跑题的(电视剧/航空串进 AI 板)踢掉,沾边/拿不准的留。是重磅度闸的话题版兄弟(水货闸管"是不是水货",话题闸管"放对板没")。

### 要解决什么问题
板块归属唯一判据是「成员源多数领域」(`source_domain == d.key`),综合源(如 IT之家只标 category:cn、domain 默认 ai)整批塞进 AI 板;短中文标题向量余弦又分不开话题(电视剧对 AI 0.479 > 真 AI 工具 0.335,纯阈值救不了)→ 跨领域内容串板,电视剧《群星》曾排 AI 板第 1。设计好的相关性闸 `allocate.assign_board`/`score.passes_relevance_gate` 是死代码没接(余弦太弱)。

### 核心设计决策 + 为什么
- **照搬重磅度闸工厂骨架**:分板后限额前、只判 heat 头部 N、fail-safe KEEP、严格 `off_topic is True` 才砍、成本闸/缓存跨板共享、记 `topic.off_dropped` 被踢清单。两道闸**串联**:话题闸先判跑题、水货闸再判水货。
- **判归属喂事件全文 + 板块画像,不用余弦**(codex 建议):余弦对短中文标题分不开话题正是根因,改 LLM 读全文判"主题属不属于这个大领域"。
- 🔴 **命门 = 只粗砍跨领域跑题,不做"领域纯度精筛"**(真数据 A/B 逼出的最关键教训):**首版判官太严**,把 geo 板"英国首相辞职/刚果埃博拉/欧洲热浪"(他国国内重大政治 + 跨国突发)当跑题误杀——它从"这条是不是落在别的大领域"滑成了"够不够格算地缘政治"。修法=收紧 `_SYSTEM` 讲死"只砍娱乐/体育/航空/刑案这种**整个落在别领域**的,本领域的商业/金融/IPO/政治/政策角度、国际板的各国国内政治+跨国突发**一律留**"+ `topic_portraits` 三板画像补边界。**宁错放别错杀**;该由水货闸杀的(个人闲聊)交水货闸,话题闸不越界。
- **配置 additive 默认关**:`topic_gate_enabled`(默认 False)+ `topic_judge_top_n`(25)+ `max_topic_judges_per_run`(全局硬顶跨板共享)+ `topic_portraits`(每板画像)。回滚 = 改 `false`。

### 现状
✅ 已上线 `config.yaml` `topic_gate_enabled: true`(commit cc42a53 实现 + ab3ec64 收紧口径)。验收三证 PASS:定向回归 17/18、独立盲考官 42 真题(命门误杀 0/22、跨领域召回 100%、复跑无抖动)、真数据同餐 A/B(被踢清单 14→7 全真跑题、geo 误杀全消失)。⚠️ 真跑端到端核 `topic.off_dropped` 日志(同 magnitude 套路)。影子工具 `deploy/perf/topic_gate_ab.py`(gitignored)。**2026-07-01 加 3 票多数票 `topic_judge_votes`**(治单判抖动把跑题货放进板;默认 3,DeepSeek 余额紧可设 1 回退单发)。

## 2.6 分板器 board_classifier(综合源读内容判板) ✅
> (2026-06-28 上线开启;详细设计原稿属内部卷宗。)
- **要解决**:50 个综合源(36氪/IT之家/大报)的 domain 标签不可信(标 ai 却发财经),按标签分板会乱跑、还挤专源名额。

## 3. 语义检索·问答翻历史 qa ✅

> **人话**:档案搜索从"字面匹配"升级成"用大白话问历史,系统召回真历史卡 + LLM 总结成回答、每句标出处",零编造。

### 要解决什么问题
旧档案搜索是纯字面匹配(前端 JS 对内联 JSON 做 `includes()`),搜"中东局势最近怎么了"=零结果。本设计让用户用大白话问历史,先语义召回真卡 + 再用 LLM 把它们总结成引用式回答,**不碰选稿/交付/events 引擎**(与重磅度闸零撞车,正是选它打头的原因)。

### 核心设计决策 + 为什么
- **铁律(pulsewire 基因,焊死)**:回答**只准用召回到的真卡内容**,每句标源(同 `{Fn}` 数字回源);召回为空/不相关 → 直说"没找到"绝不编;LLM 挂/DB 没起 → 报"不可用"不静默产空。
- **索引档案卡向量**:给 `summaries` 加 `card_vec vector(1024)`(= `embed_passage(headline+tldr+insight)`),一次性回填历史 + 每天 summarize 后增量写。比复用 item 标题向量准(item 标题 ≠ 发布 headline,且含大量没发布的噪声)。
- 🔴 **jina-v3 须分调 `embed_query`/`embed_passage`**(不是 `embed(task=)`,现 `embed()` 走默认 text-matching task → 召回错位);问题用 query 前缀、回填卡用 passage 前缀,**必须配对**。召回函数是**新写** `recall_cards_by_vector`(查 `summaries.card_vec` 返回 Summary,与旧 `recall_by_vector` 查 item 标题向量返回 Item 毫无交集——不是复用)。
- 🔴 **零编造的真漏洞(RAG 头号失效模式)**:运行时校验只挡"`used` 填了不存在的卡号",**挡不住** LLM 填合法卡号 `[1,2]` 但 answer 写的事实来自训练数据/借题发挥。这条**运行时拦不住、只有考官能逮** → 验收必须有专项考官任务(喂"相关但细节有出入"的卡,逐句把答案事实追溯回被引卡文本)。
- 🔴 **旧系统数据隔离**:`summaries` 无来源字段、`interest_key` 是 hash(非 "ai"/"bio" 字符串),按板块筛不准 → 迁移加 `produced_by='pulsewire'` 字段、QA 召回硬过滤(误把旧系统回填数据当 pulsewire 历史召回,踩过坑)。
- **形态不变**:每次提问临时起一次 Python 进程算(无常驻服务),符合现有"无 server"静态站;字面搜保留并行,"问"是新增模式不替换。Electron spawn 前查 Postgres + 给"思考中"态(冷启加载模型 ~2-3s)。

### 现状
✅ Phase 1+2 全落地 + 独立验收 PASS(2026-06-23/24)。迁移 0010(`card_vec` + `produced_by`)、`qa/engine.py` + `pulsewire ask` CLI、card_vec 增量已接日报管道、桌面 App 档案页问答面板已接线、RUBRIC 补 qa 维。独立盲考官含陷阱题**零编造 PASS**。⚠️ "80+ 天历史"是误数(DB 现存 ~9 天 summaries,随天数自然长)。

## 4. 事件线 / 在追(跨天盯梢) ✅

> **人话**:让同一件事(OpenAI 上市、伊以冲突)能**跨天追着看演变**,而不是每天一张互不相干的快照。这是 pulsewire 区别于"更靠谱的热搜榜"的唯一硬核理由。

### 要解决什么问题
「簇」(cluster)只合同一条新闻的不同版本(≤3 天、措辞高度相似);缺的是把**措辞差很大、但属于同一个故事**的多个簇串成一条「事件线」(thread,跨几周、剧情和措辞都在变)。现有"相似度合并"门槛太高(宁漏勿误)够不着剧情演进。

### 核心设计决策 + 为什么
- **方案 B+A(已拍板)**:**A 认主角**(给每簇抽「事件主体」短语如 `OpenAI-IPO` 做预过滤,省钱防乱)+ **B AI 判官**(把 A 缩到很少几条的候选线 + 簇 headline 交 LLM 判"接哪条线还是新开")。不选纯 A(太笨,"OpenAI 上市"和"OpenAI 发新模型"会混);不选放宽现有合并(方案 C,会动已上线验收的当天去重,**不碰**)。
- **挂载单元用「簇」不用「单条新闻」**:层级干净(item→cluster→thread 三层)。
- **设计成可重放(派生数据非既成事实)**:`thread_clusters` 记全判定输入痕迹(subject/link_reason/confidence/run_id),`pulsewire threads --rebuild` 从归档重放重建——判定逻辑日后改不留死账。
- **同故事裂线已根治**:原 A 用纯词法匹配,同故事不同措辞会裂成多条线(伊朗局势裂成 peace/war/conflict)→ 升级为「词法 ∪ 语义(embedding)」召回(`threads/recall.py`,开关 `threads.semantic_match` 默认开);实测同故事余弦 0.70~0.99、不同故事 0.44~0.53(不乱合)。
- **失败必降级不拖垮日报**:threads 站自吞异常(事件线是增强,主交付优先);flash 偶发空返回 ~10% → 退避重试 3 次,坏 JSON 条目优雅降级跳过。
- **GitHub 涨星排序(独立子集先上)**:事件线"跨天对比"最简版,不用 LLM 不用新表,查 `item_timeline.stars` 最近两次快照算 delta 降序排,只一份快照退化为按总星(兜底)。
- **热度反哺**:属多天在追线的日报卡加「⟿ 持续关注 · N天」徽标;候选簇属多天在追线 → 规则分 +`weights.thread`(默认 0.1)。前端「在追」视图露出门槛默认跨 ≥2 个不同日期(防刷屏)。

### 现状
✅ step 0–6 全完成上线(2026-06-14)。⛔ 不碰现有 dedup 去重阈值/窗口(方案 C),避免污染已验收的当天日报。

## 5. v2 板块扩展(4→7 板块) 📐

> **人话**:把 pulsewire 从"私人 4 板块日报"扩成"开源 + 勾选式多板块菜单"——菜单做厚,每人盘子小,谁 clone 谁勾自己要的几块。

### 要解决什么问题
现状 4 板块(AI / GitHub / 生物医疗 / 国际局势)是绑死某个人兴趣的私人配置。v2 要扩成开源他人可复用、可勾选定制的多板块日报,且**质量优先:宁可少而精,不要多而水**。不做订阅/账号/服务器,各人跑各人的。

### 核心设计决策 + 为什么
- **7 大板块骨架(锁死)**:AI / 开源与开发者 / 科技(商业·前沿·网安·生物医疗·中国科技) / 财经 / 国际局势 / 文体娱 / 社会民生。现 GitHub→并入②,生物医疗→并入③小类。
- 🔴 **核心原则:深源进、大白话出(用户 2026-06-20 纠正,锁)**:源**要深要权威**(arXiv/实验室官方/Nature/bioRxiv),**不为"怕看不懂"找浅源**;翻人话是 summarize 层的职责(已有"外行可读铁律")。太小众不值得占版面的(如纯数学证明)由**重磅度闸**筛掉,不靠避开深源。
- **跨板块一稿两投(甲·够格才两投)**:一条新闻真同时是两个板块的大事 → 各写各的角度(同一件事两副眼镜,不算重复);只是沾边 → 归最强的那个板。机制 = 复用"同事件"判定 + 一道"够不够格在第二板也算大事"的闸,与**重磅度闸同类**(两投闸复用其判官骨架)。
- **勾选下探到小类 + 显示自适应**:勾 1 个小类 → 升级为板块标题(不挂大类壳);勾 ≥2 → 显示为大类板带子目录。含义:小类必须是可独立成板的一等单元(自带源 + 身份)。
- **分两步干**:先**改造**(科技/开源扩源,50 源已双刀验证可长期抓取)→ 再**新增**(财经/文体娱/社会民生)。源策略 = 乙(我配用户审);新板块源量/精度要求低于 AI(151 源不必复制到每块)。

### 现状
📐 主线 A 板块扩展**方向已锁、未动生产代码**(改板块结构属架构级,开工前先递方案)。主线 B 第一优先 4 条进度:重磅度闸 ✅ 上线、语义问答 ✅ 上线、GitHub 榜 v2 🚧(Phase A+B1 已落地)、突发大事单独推 ⬜ 未动。上线开关已完成(2026-07 开源,MIT)。
